# FileMaker Coding Conventions

Source: [filemakerstandards.org/code](https://filemakerstandards.org/code)

> **CRITICAL — Hard Tabs for Indentation**
>
> All indentation inside calculations and expressions MUST use hard tab characters (ASCII 0x09), never spaces. This applies to `Let()` variables, `Case()` branches, `List()` arguments, and any nested function content that appears inside CDATA blocks in fmxmlsnippet output. AI-generated code must use literal `\t` tab characters for calculation indentation. In the FileMaker calculation dialog, use Option-Tab (Mac) or Ctrl-Tab (Win) to insert a tab.
>
> This rule applies to **calculation expressions only** — it does not govern HR script step indentation (which uses spaces for visual nesting in the editor).

These conventions apply to all FileMaker code generated by AI in this project — both scripts and calculation code. When any convention is ambiguous, prefer clarity and readability over brevity.

## Naming Conventions

### Variables

| Scope                    | Prefix                     | Example            |
| ------------------------ | -------------------------- | ------------------ |
| Script-local             | `$`                        | `$invoiceTotal`    |
| Global                   | `$$` (ALL_CAPS with dots)  | `$$USER.ACCOUNT`   |
| Calculation-scoped (Let) | `~`                        | `~lineTotal`       |
| Private global           | `$$~` (ALL_CAPS with dots) | `$$~PRIVATE.CACHE` |

- Use camelCase after the prefix for all variables
- Boolean variables use present-tense verbs or adjectives: `$hasReturns`, `~isTrailing`, `$isValid`

### Tables

- Single word, TitleCase: `Invoice`, `LineItem`, `Customer`

### Fields

| Type          | Convention             | Example                    |
| ------------- | ---------------------- | -------------------------- |
| Regular       | camelCase              | `firstName`, `totalAmount` |
| Summary       | camelCase, descriptive | `summaryCountInvoices`     |
| Global        | ALL_CAPS               | `GLOBALFIELD`              |
| Primary key   | `id`                   | `id`                       |
| Foreign key   | `id_Tablename`         | `id_Customer`              |
| Unstored calc | `unstored_` prefix     | `unstored_FullName`        |

### Layouts

- Format: `Tablename: Layout Name (form|list)`
- Portal objects: `portal.layoutObjectName`
- Example: `Invoice: Detail (form)`, `Customer: List (list)`

### Scripts

- Title Case with spaces: `Send Confirmation Email`
- Parameters indicated with parens and semicolons: `Send Email ( to ; subject ; body )`
- Folder organization (see Scripting & Organization section below)

### Custom Functions

- TitleCamelCase: `FormatPhone`, `ObjectID`, `CardWindowHeight`
- Private (used only by other custom functions): prefix with `~`: `~PrivateHelper`

## Calculation Code

### Let() Formatting

Opening and closing brackets on their own lines. Each variable on its own line, indented with a hard tab:

```
Let ( [
		~subtotal = invoice_LineItems::unitPrice * invoice_LineItems::quantity ;
		~discount = ~subtotal * $discountRate
	];

	~subtotal - ~discount
)
```

- **Hard tabs** (ASCII 0x09, not spaces) for all indentation inside calculations — this is non-negotiable. In the FileMaker calculation dialog, use Option-Tab (Mac) or Ctrl-Tab (Win). AI-generated code MUST use literal tab characters, never spaces, for calculation indentation.
- Space after the semicolon when the function continues on the next line (continuation clue)
- No space before the semicolon when a line is complete

### Operators

- Spaces before and after comparison and assignment operators: `$foo = bar`, `~total > 0`
- Preferred comparison symbols: `≤` `≥` `≠` over `<=` `>=` `<>`
- Exception — no extra space when incrementing inline: `$counter + 1`

### String Concatenation

- Single space on both sides of `&`: `"Hello " & $name`
- Exception — carriage returns on the same line: `Get ( AccountName ) &¶& Get ( CurrentTimeStamp )`
- For multi-line concatenations, consider `List()` for readability:

```
List (
	$firstName ;
	$lastName ;
	$email
)
```

### Boolean Values

- `True` and `False` are the preferred boolean literals for code readability — FileMaker evaluates them as 1 and 0
- Boolean variable names use present-tense verbs or adjectives: `$isComplete`, `~hasValue`

### Function Spacing

- Space between function name and opening paren is optional but keep it consistent within a calculation
- Space after each semicolon separator: `Left ( $text ; 3 )` not `Left($text;3)`
- This supports cross-platform double-click selection of arguments

### No Alignment Padding

Do not add extra spaces to align values across lines. FileMaker does not render code in a monospace editor where column alignment is meaningful — the padding adds noise without benefit.

```
// Wrong — padded for alignment
JSONSetElement ( "{}" ;
	[ "label"     ; $label   ; JSONString ] ;
	[ "timestamp" ; $ts      ; JSONString ]
)

// Correct — single spaces only
JSONSetElement ( "{}" ;
	[ "label" ; $label ; JSONString ] ;
	[ "timestamp" ; $ts ; JSONString ]
)
```

This applies to all multi-argument functions: `JSONSetElement`, `List`, `Case`, `Let`, etc.

### Dynamic Field References

When a field name is passed as a **string parameter** to a step or function (e.g., `Set Field By Name`, `GetField`), prefer wrapping it in `GetFieldName()` to make the reference rename-safe:

```
# Preferred — rename-safe
Set Field By Name [ GetFieldName ( Invoices::Status ) ; "Sent" ]
GetField ( GetFieldName ( Invoices::Status ) )

# Brittle — breaks silently on field rename
Set Field By Name [ "Invoices::Status" ; "Sent" ]
GetField ( "Invoices::Status" )
```

Literal strings are acceptable when the field name is genuinely dynamic (e.g., user-selected at runtime or iterated from `FieldNames()`). See `agent/docs/knowledge/field-references.md` for comprehensive patterns and edge cases.

## Control Structures

### If Statements

- Test argument starts on the same line as `If`
- Complex multi-condition tests split across lines with each condition indented
- Comment `// Else` before each `Else` branch for readability in script context

```
If [ $status = "Active" ]
  // ... steps
Else If [ $status = "Pending" ]
  // ... steps
// Else
Else
  // ... steps
End If
```

### Case Statements

- Each test and result on its own line
- Result indented from its test
- Separate each test/result pair with a blank line
- Boolean operators (`or`, `and`) at the start of the continuation line, not the end of the previous line:

```
Case (
	~value = 1 ;	"One" ;

	~value = 2
	or ~value = 3 ;	"Two or Three" ;

	"Other"
)
```

## Commenting

- Inline comments use `//`
- Use `MARK: SectionName` (disabled `# (comment)`) at the top of logical sections within a long script (similar to code folding markers) — no `//` prefix
- Comment the _why_, not the _what_ — avoid restating what the step obviously does
- In Case/If branches, use `// Else` as a visual separator before Else steps
- Script-level comments describe purpose, expected context, parameters, and return values

### HR format comment syntax

FileMaker's `# (comment)` step has two states — enabled and disabled — and both appear in HR format. The syntax convention is:

| HR syntax | Step state | XML | Use for |
|-----------|-----------|-----|---------|
| `# text` | Enabled (`enable="True"`) | `<Step enable="True" id="89" name="# (comment)"><Text>text</Text></Step>` | Visible annotations, header lines |
| `#// text` | Disabled (`enable="False"`) | `<Step enable="False" id="89" name="# (comment)"><Text>MARK: SectionName</Text></Step>` | Section markers, structural dividers — use `MARK: SectionName` without any `//` prefix |
| `#` | Enabled, blank | `<Step enable="True" id="89" name="# (comment)"/>` | Visual blank line between sections |

The `//` prefix on a disabled comment is a visual convention borrowed from other languages — it signals "this step is off" in the same way `//` means a commented-out line. `// MARK:` headers always use the disabled form `#//` so they appear greyed out in the Script Workspace and do not clutter the active comment layer.

This distinction matters when HR code is pasted into FileMaker or converted by the webviewer's HR→XML converter: `# text` and `#// text` must produce different `enable` attribute values.

### Script documentation structure

Every script follows this documentation pattern at the top, written entirely as enabled `# (comment)` steps. Each line of the header is a separate step.

**Header format — JSON parameters/returns:**

```
# Script description
# 
# PARAMETERS {JSONObject}
#    key1 {JSONString} description
#    key2 {JSONObject} description
#    key2.keyA {JSONNumber} description
#    key2.keyB {JSONBoolean} description
#    key3 [JSONArray] of {JSONObjects} description
#    key3.[].keyC {JSONString} description
#    key3.[].keyD {JSONString} description
# 
# Example
# JSONSetElement ( "{}" ;
#    [ "key1" ; "value" ; JSONString ] ;
#    [ "key2.keyA" ; 1 ; JSONNumber ] ;
#    [ "key2.keyB" ; True ; JSONBoolean ] ;
#    [ "key3.[+].keyC" ; "C" ; JSONString ] ;
#    [ "key3.[:].keyD" ; "D" ; JSONString ] ;
# )
# 
# RETURNS {JSONObject}
#    success {JSONBoolean} true
#    message {JSONString}
# OR
#    success {JSONBoolean} false
#    errorCode {JSONNumber} FM error code (positive) or custom code (negative)
#    message {JSONString}
# 
#    Error codes: -1 invalid parameter, -2 not found  (example — define per script)
# 
```

**Header format — non-JSON (FileMaker field types):**

```
# Script description
# 
# PARAMETERS {Text}
#    $theVariableName - variable description
# 
# RETURNS {Number}
#    description of the return value
# 
```

**Error code convention:**

Scripts that return JSON always include `errorCode` in the failure shape:

| Range | Meaning | Example |
|-------|---------|---------|
| Positive integer | Native FileMaker error — the value returned by `Get ( LastError )` | `401` record missing, `301` record locked |
| Negative integer | Custom scripting error — validation failure or logical condition the script detected | `-1` invalid parameter, `-2` not found |

Use negative numbers starting at `-1` for script-defined failures, counting down as needed. Document the codes used in the script's header under the error code comment line.

**Type notation:**
- Scalar: `{JSONString}`, `{JSONNumber}`, `{JSONBoolean}`, `{JSONObject}`, `{Text}`, `{Number}`, `{Date}`, `{Timestamp}`
- Array: `[JSONArray] of {JSONObjects}`
- Array element key path: `arrayKey.[].childKey {type}`
- In examples: `[+]` appends a new element; `[:]` replaces all elements

**XML — each `#` line is its own step:**
```xml
<Step enable="True" id="89" name="# (comment)">
  <Text>Script description</Text>
</Step>
<Step enable="True" id="89" name="# (comment)"/>
<Step enable="True" id="89" name="# (comment)">
  <Text>PARAMETERS {JSONObject}</Text>
</Step>
<Step enable="True" id="89" name="# (comment)">
  <Text>   key1 {JSONString} description</Text>
</Step>
```

Blank `#` lines use the self-closing form `<Step enable="True" id="89" name="# (comment)"/>`.

**Section markers** (`MARK:`) use the *disabled* form so they appear greyed out in Script Workspace:
```xml
<Step enable="False" id="89" name="# (comment)">
  <Text>MARK: SectionName</Text>
</Step>
```

## Scripting & Organization

### Script Naming

- Title Case with spaces: `Process Invoice`, `Validate Line Items`
- Include parameters in the name when it aids discoverability: `Navigate To Layout ( layoutName )`
- Prefix with a functional area when scripts are shared: `Email: Send Confirmation`

### Folder Structure (baseline)

```
Application/
  Startup
  Shutdown
  Menus
  Shared/          ← solution-wide utility scripts
User Interface/
  <Layout Name>/   ← one folder per layout, named to match
```

- Scripts specific to a single layout go in that layout's folder
- Reusable scripts shared across layouts/contexts go in `Shared/`
- Extend the structure as needed — this is a minimum baseline

### Script Step Conventions

- Use `Exit Script [ Result: ... ]` to return values, not global variables, wherever possible
- Always `Commit Records/Requests` before navigating away from a record context if edits may have occurred
- Error capture (`Set Error Capture [ On ]`) should be paired with explicit error checking immediately after the risky step
- Comment the expected script parameter format at the top of any script that accepts parameters

## Empty Strings / Null Values

- Use `""` for an empty string — never a space `" "`
- To test for empty: `IsEmpty ( field )` is preferred over `field = ""`
- Distinguish between truly empty and a field containing only whitespace when it matters: `IsEmpty ( Trim ( field ) )`

---

## Reserved Words

Avoid using FileMaker reserved words as field, variable, table, or script names. Common ones to avoid:

`Status`, `Error`, `Name`, `Date`, `Time`, `Count`, `List`, `Max`, `Min`, `Sum`, `Average`

Disambiguate with context: `invoiceDate`, `customerName`, `lineItemCount`
